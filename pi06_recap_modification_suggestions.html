pi0.6 / RECAP 开源仓库修改建议

生成时间：2026-06-02 ｜ 目标：将当前仓库从“RECAP 风格训练雏形”推进到“可用于离线验证与后续真机 HITL-RL 的工程版本”
总体判断：当前仓库已经完成了 RECAP 的部分工程入口，包括 advantage 字段、advantage prompt tokenization、prefix 注入、RECAP 双路 loss、reward/return/advantage 工具和迭代训练 CLI 骨架。但它还没有完成论文中的完整 π0.6 / π*0.6 RECAP 闭环。最关键缺口是：完整视觉语言 value function、value training loop、value inference 到 advantage label 的写回、真实 rollout、human intervention 回调、以及 Algorithm 1 的自动化编排。
评估修订：本文档的总体方向与论文一致，但需要注意两个细节。第一，原先代码里的 positive_quantile=0.4 如果直接用于 np.quantile() 后再执行 advantage >= threshold，会让约 60% 样本变成 positive；若目标是论文 fine-tuning 常用的约 40% positive，应使用 0.6 分位阈值，或把参数改名为 positive_fraction 并在内部转换为 1 - positive_fraction。第二，规则版 value proxy 只是离线工程打通方案，不等价于论文中的视觉语言 distributional value function。
建议策略：不要一开始追求完整复现论文。更现实的路线是先做“离线版 RECAP”：用已有成功/失败 episode 训练 value model 或先用规则 value proxy，生成 advantage_indicator，跑 pi05_recap 训练，验证是否优于普通 π0.5 SFT。离线版本跑通后，再接机器人 rollout 和人类干预。

2. 与论文相比的关键缺口

2.1 缺少真正的 π0.6 模型规模与结构

论文中的 π0.6 使用更大的 backbone 和 action expert，例如 Gemma 3 4B 与 860M action expert。当前仓库仍主要基于 openpi / π0.5，配置中默认是 gemma_2b 和 gemma_300m，RECAP 配置也加载 pi05_base checkpoint。因此当前更准确地说是：

π0.5 + RECAP-style advantage conditioning

而不是完整论文级 π0.6 / π*0.6。

2.2 缺少完整 vision-language value function

论文要求训练一个语言条件的 distributional value function，输入图像、机器人状态和语言指令，输出 201 个 value bins，用于估计状态价值和动作 advantage。当前只有一个轻量 DistributionalValueHead 和几个工具函数，缺少：

  • 完整 value model backbone；
  • 图像 + 语言 + state 的 value forward；
  • cross-entropy value loss；
  • value checkpoint 保存与加载；
  • value inference pipeline。

2.3 缺少 value inference 到 advantage label 的写回流程

当前 assign_advantage_labels() 只是接收外部 advantages，然后按 quantile 打标签。它不会自动用 value model 计算 advantage，也不会把 advantage_indicator 写回 LeRobot dataset。

此外，当前 quantile 参数语义需要修正：如果代码使用 threshold = np.quantile(advantages, positive_quantile)，再把 advantages >= threshold 标为 positive，那么 positive_quantile=0.4 实际会产生约 60% positive 样本。论文 Appendix F 描述 fine-tuning 通常设置约 40% rollout 数据为 positive，T-shirt/shorts 任务甚至约 10%。因此建议明确区分“分位阈值”与“positive 样本比例”。

2.4 缺少 human intervention 的真实闭环

代码中虽然有 is_human_intervention 字段，但还没有接入 SpaceMouse、键盘、WebSocket、遥操作手柄等人类接管回调。并且当前打标签逻辑也没有强制将 human intervention 动作标为 positive advantage。

2.5 缺少真实 rollout / 仿真 rollout 采集

论文中的 RECAP 依赖真实机器人自主执行、失败/成功标注、人类在线修正和 episode 存储。当前 scripts/recap_train.py 只是流程日志，缺少：

  • env.reset() / env.step(action) adapter；
  • policy server / robot runtime 调用；
  • success / done / timeout 标注；
  • episode recorder；
  • 安全停止机制。

2.6 缺少 Algorithm 1 自动化编排

论文 Algorithm 1 是完整闭环：

collect data -> train value -> compute advantage -> fine-tune VLA -> repeat

当前 CLI 只是记录流程顺序，没有实际调用子过程。

2.7 RECAP loss 只覆盖 flow matching，缺少完整离散 token / sub-task 目标

论文中 π0.6 同时包含高层 sub-task 文本、FAST 离散动作 token 和连续 flow matching action。当前 RECAP loss 主要包裹 flow matching loss，没有完整覆盖论文中的离散动作 likelihood、高层 sub-task 预测和 KI 联合训练目标。

3. 修改目标分层

版本 目标 适用场景 建议优先级
V0：当前版本 advantage-conditioned VLA 训练雏形 代码验证、smoke test 已具备
V1：离线 RECAP 用已有 episode 训练/估计 value，写回 advantage，训练 VLA 你的试管架数据、ACT/π0.5 离线评估数据 最高
V2：仿真 RECAP 接 MuJoCo/Isaac Sim rollout，自动生成 success/failure 先不冒真机风险，验证闭环 高
V3：真机 HITL-RECAP 接真实机器人、人类干预、安全停机、episode recorder 真实试管架抓取、插槽、力控任务 中高
V4：论文级 π0.6 更大 backbone、完整 KI、sub-task、discrete/continuous 联合训练 长期研究复现 低于 V1/V2/V3

4. 第一阶段：补齐离线 RECAP，建议优先实现

阶段 1.1：定义 RECAP episode 标准格式

目标是把已有 LeRobot / rosbag / eval episode 转成统一结构，后续 value training、advantage 计算、VLA 训练都基于这个格式。

建议字段：

{
  "episode_id": "xxx",
  "task": "pick test tube rack and insert into slot",
  "success": true,
  "timeout": false,
  "frames": [
    {
      "t": 0,
      "images": {...},
      "state": [...],
      "action": [...],
      "is_human_intervention": false
    }
  ],
  "metadata": {
    "source": "demo/eval/rollout/intervention",
    "max_episode_length": 500,
    "operator": "xxx"
  }
}

建议新增文件：

  • src/openpi/training/recap_episode_io.py
  • scripts/convert_lerobot_to_recap_episodes.py
阶段 1.2：先实现规则版 value proxy

在完整 value model 没有完成之前，不要卡住。可以先用成功/失败和剩余时间构造 value proxy，用于打通离线数据标注、写回和 VLA 训练链路：

成功 episode:
V(t) = - remaining_steps / max_episode_length

失败 episode:
V(t) = -1.0 或者根据失败发生时刻设置更低 value

这样可以先生成 advantage_indicator，跑通 RECAP VLA 训练链路。但这只是工程 proxy，不是论文中的视觉语言 value function：论文的 value function 输入图像、机器人状态和语言指令，输出 201-bin distributional value，并用 cross entropy 训练。

建议新增文件：

  • src/openpi/training/recap_value_proxy.py
  • scripts/label_recap_advantage.py
阶段 1.3：修正 human intervention 的 positive 标签逻辑

论文中 human correction 动作应强制视为 positive。当前 assign_advantage_labels() 没有处理这一点，建议修改为：

labels = episode_advantages >= threshold
labels = labels OR episode.is_human_intervention

同时建议保留 label_source 字段，用于区分 value positive、human positive、manual positive。

建议同时修正 quantile 参数命名，避免把“positive 比例”和“分位阈值”混用：

# 如果用户配置的是目标 positive_fraction，例如 0.4
threshold_quantile = 1.0 - positive_fraction
threshold = np.quantile(task_advantages, threshold_quantile)
labels = advantages >= threshold
阶段 1.4：写回 LeRobot dataset

需要把以下字段写回训练数据：

  • advantage_indicator
  • use_advantage
  • is_human_intervention
  • success / episode_return / value_target_bin，用于后续 value model 训练

建议脚本：

PYTHONPATH=src python scripts/label_recap_advantage.py \
  --input-episodes /path/to/episodes \
  --output-lerobot /path/to/lerobot_recap \
  --positive-quantile 0.4

5. 第二阶段：实现真正的 value function

阶段 2.1：实现 ValueModelConfig

建议不要只用一个 DistributionalValueHead。应新增完整 value model：

  • 复用 Pi0/PaliGemma 的图像和语言 encoder；
  • 去掉 action expert；
  • 增加 value head，输出 [B, value_bins]；
  • 输入 observation + language prompt；
  • 输出 distributional value logits。

建议新增文件：

  • src/openpi/models/recap_value_model.py
  • src/openpi/models/recap_value_config.py
阶段 2.2：实现 value training loop

训练目标为 distributional cross entropy：

loss = CE(value_target_bin, value_logits)

建议新增脚本：

PYTHONPATH=src python scripts/train_recap_value.py \
  --config-name recap_value_pi05 \
  --dataset /path/to/recap_lerobot \
  --exp-name value_v1
阶段 2.3：实现 value inference 与 advantage 计算

需要对每个 timestep 计算：

A(o_t, a_t) = sum(r_t ... r_{t+n-1}) + V(o_{t+n}) - V(o_t)

然后根据任务内 quantile 或固定阈值生成 advantage_indicator。

注意：论文中是任务相关阈值，建议先按 task_id 或 prompt 分组计算 quantile，不要全数据混在一起。

6. 第三阶段：补齐迭代训练 orchestration

当前 scripts/recap_train.py 只是日志骨架。建议改成真正执行以下流程：

for iteration in range(K):
    1. load demos + previous rollouts
    2. train value model from pretrained value checkpoint
    3. run value inference and compute advantage
    4. write advantage_indicator back to dataset
    5. fine-tune VLA from base checkpoint, not last checkpoint
    6. evaluate policy
    7. collect new rollout data
    8. save iteration report

建议输出每轮报告：

  • episode 数量；
  • 成功率；
  • 平均完成时间；
  • advantage positive 占比；
  • human intervention 占比；
  • value loss；
  • VLA loss；
  • eval success rate；
  • 下一轮是否继续。

7. 第四阶段：接仿真或真机 rollout

7.1 先接仿真环境

建议先接 MuJoCo 或 Isaac Sim，不要直接上真机闭环。最小接口如下：

class ReCAPEnv:
    def reset(self) -> dict:
        ...

    def step(self, action) -> tuple[dict, float, bool, dict]:
        ...

    def success(self) -> bool:
        ...

    def close(self):
        ...

7.2 再接人类干预

建议定义统一 intervention callback：

class InterventionCallback:
    def get_action(self, obs) -> np.ndarray | None:
        # None: 不接管
        # action: 人类接管动作
        ...

干预时 episode recorder 需要记录：

  • policy action；
  • human action；
  • 最终执行 action；
  • is_human_intervention；
  • 干预开始/结束时间；
  • 干预原因：危险、失败趋势、卡住、误抓、插入失败等。

7.3 真机安全要求

真机 HITL-RL 必须加安全层：速度限制、关节限位、末端工作空间限制、碰撞检测、力/力矩阈值、急停、超时停止、动作平滑、gripper 保护逻辑。不要让 VLA 直接裸控机器人。

8. 针对你的试管架任务的落地方案

你的任务是试管架抓取并放入检测设备卡槽，容差可能只有 2–4 mm。这个任务不适合纯 VLA 端到端硬上。建议把 RECAP 用在中上层动作策略改进，而插入阶段仍保留传统视觉伺服和力控保护。

阶段 建议控制方式 RECAP 的作用
接近试管架 VLA / ACT / 视觉粗定位 学习不同初始位置下的接近策略
抓取试管架 VLA + gripper 规则保护 减少抓偏、抓空、从上方误抓
移动到设备附近 VLA / FSM / IK 提高路径流畅性和稳定性
对准卡槽 传统视觉伺服 / 6D pose / wrist camera 可学习粗对准策略，但不建议完全替代伺服
插入 视觉伺服 + 力控 + impedance / admittance RECAP 可用于学习失败恢复策略，但底层必须有安全控制

9. 推荐开发顺序

  1. 先修正 assign_advantage_labels()：明确 quantile/positive_fraction 语义，并让 human intervention 强制 positive。
  2. 实现离线 episode 格式：把 demo/eval 数据统一成 RECAP episode。
  3. 实现规则版 value proxy：先不训练 value model，基于成功/失败和剩余时间生成 advantage。
  4. 写回 LeRobot dataset：补齐 advantage_indicator、use_advantage、is_human_intervention。
  5. 跑 pi05_recap 训练：对比普通 pi05 SFT。
  6. 建立评估报告：success rate、完成时间、抓取偏差、插入偏差、失败类型。
  7. 再实现真正 value model：用完整视觉语言 value function 替代规则 proxy。
  8. 最后接仿真/真机 rollout：形成闭环。

11. 最小可运行版本验收标准

V1 离线 RECAP 验收标准：
  1. 能够从已有成功/失败 episode 生成 reward、return、advantage。
  2. 能够写回 advantage_indicator 到训练数据。
  3. pi05_recap 可以正常训练，不报 shape / JIT / dataloader 错误。
  4. 同一批数据下，与普通 π0.5 SFT 对比，至少在 eval 成功率、平均完成时间、失败类型数量中的一个指标上有改善。
  5. 输出 HTML 评估报告，包含每轮训练、数据标签分布、成功率和失败案例。
V2 仿真 RECAP 验收标准：
  1. 仿真环境可以自动 rollout。
  2. episode recorder 可以保存完整 observation/action/success/failure。
  3. 每轮自动完成：collect → label → train → eval。
  4. 至少 2 轮迭代后，仿真 success rate 或 throughput 有提升。
V3 真机 HITL-RECAP 验收标准：
  1. 支持人类接管，并正确记录 intervention action。
  2. human intervention 样本被强制标为 positive。
  3. 真机 rollout 有安全限幅和急停。
  4. 每轮训练后，任务成功率提升或失败模式减少。

12. 结论

当前仓库不是完整论文复现，但方向是对的。它已经把最核心的“advantage conditioning 接入 VLA”打通了第一步。需要优先修正的是 advantage 标签生成语义，尤其是 quantile 阈值和 positive 样本比例不要混用。后续不要优先追求更大的 π0.6 模型，而应该优先补齐：

  1. 离线 episode 标准化；
  2. value / advantage 计算；
  3. advantage label 写回；
  4. value model 训练；
  5. 迭代训练 orchestration；
  6. 仿真/真机 rollout；
  7. human intervention 和安全闭环。

最务实的下一步：先实现 V1 离线 RECAP。只要能证明同一批数据下 pi05_recap 比普通 pi05 SFT 更稳、更快、更少失败，这个仓库就有继续投入的价值。
